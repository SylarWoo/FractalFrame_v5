这说明它已经拿到了“总数/目标数”，但图表没有进入“加载完成 / ready”状态。也就是说，当前卡在：

## 先给结论

刚才“它那个首页可以加载 300 多万”，大概率是因为首页走的是**静态全量 JSON / mock datafeed / 一次性喂图表**那条链。

你自己这个页面现在走的是：

```txt
StoreV5 仓库
  ↓
DuckDB 查询接口
  ↓
fetch JSON
  ↓
前端转换成 KLineCharts 数据
  ↓
KLineChart Pro datafeed 状态
```

这条链多了后端查询、JSON 序列化、网络传输、前端转换、loading 状态收口。任何一个环节没有正确返回/回调，页面就会一直显示“加载中”。

---

## 最可能的 3 个原因

### 1. 你一次性请求了 319 万 rows，JSON 传输和前端转换卡住

3,190,000 根 M1 如果从 DuckDB 查出来，再转成 JSON，再传给浏览器，再转成 KLineCharts 的 `timestamp/open/high/low/close/volume`，这会非常重。

首页能加载，不代表 StoreV5 API 全量 JSON 加载也能顺滑。因为首页可能是：

```txt
静态文件 / mock / 已经格式化好的数组
```

而现在是：

```txt
DuckDB 查 parquet
  ↓
Python rows -> JSON
  ↓
HTTP 返回巨大 JSON
  ↓
浏览器 JSON.parse
  ↓
React / datafeed 再转换
```

KLineCharts 本身可以吃大量数据，但**不要把“数据传输链路”也当成图表内核性能**。

---

### 2. KLineChart Pro 的 datafeed 没有正确调用“加载完成回调”

你截图左下角一直是：

```txt
加载中 3,192,046
```

这很像 datafeed 已经开始 loading，但没有正确触发完成状态。也就是可能有这种问题：

```txt
getHistoryData / loadData 开始
  ↓
请求成功
  ↓
数据也拿到了
  ↓
但没有调用 callback / 没有 resolve / 没有设置 hasMore / 没有更新 loading=false
```

所以图上能看到一部分 K 线，但状态还卡在 loading。

这个要查你前端真正接入 KLineChart Pro 的 datafeed 文件。现在我搜索 V5 仓库，能看到 mock datafeed 文件 `frontend/src/datafeed/createMockKLineChartProDatafeed.ts`，说明项目早期有 mock datafeed 入口。

但我没在搜索结果里直接看到 StoreV5 datafeed 文件名，说明现在你这个页面的数据接入可能散在别的组件里，或者还不是一个干净的 `createStoreV5KLineChartProDatafeed.ts`。

---

### 3. 你现在可能还在“一次性全量加载”，而不是分页加载

你当前状态写着加载 3,192,046，说明它很可能试图把全量 M1 一口气送进 chart。

这个阶段应该先不要一口气全量。正确做法是：

```txt
首次加载最近 5,000 ~ 20,000 根
  ↓
图表 ready
  ↓
用户左拖，再按窗口加载更早数据
```

而不是：

```txt
首次打开页面直接加载 3,190,000 根
```

首页压测可以一次性全量，因为它就是为了压测。
正式页面不应该默认一口气全量。

---

## 现在要怎么查

让执行者不要先乱改，先加日志查这几个点：

### 1. Network 查 query 接口

打开浏览器 DevTools → Network，看 StoreV5 query 请求：

```txt
/api/market-data/v1/store-v5/query?symbol=XAUUSDm&timeframe=M1...
```

看三个东西：

```txt
Status 是不是 200
Response size 多大
请求耗时多久
```

如果 response size 是几百 MB，那就是全量 JSON 传输太重。

---

### 2. Console 查 datafeed 是否完成

在你自己的 datafeed 里加：

```ts
console.log('[StoreV5Datafeed] request start', params)
console.log('[StoreV5Datafeed] response rows', rows.length)
console.log('[StoreV5Datafeed] converted rows', data.length)
console.log('[StoreV5Datafeed] callback done')
```

如果能看到 rows 3192046，但没有 callback done，那就是 datafeed 收口问题。

如果连 response rows 都没有，那就是 query 接口/JSON 传输卡住。

---

### 3. 后端查 query 是否真的返回

在 `/store-v5/query` 后端加日志：

```txt
query start
duckdb rowsCount
json build start
json build done
response sent
```

如果 DuckDB 很快，但 JSON build 很慢，就是 Python JSON 序列化问题。

## 正确修复方向

### 第一刀：正式页面默认只加载最近窗口

不要默认加载 3,190,000。

正式页面首次加载：

```txt
limit = 10000
```

或者：

```txt
limit = 20000
```

查询：

```txt
symbol=XAUUSDm
timeframe=M1
mode=direct
limit=10000
```

图表 ready 后，左拖再加载历史。

---

### 第二刀：把全量加载入口和正式页面入口分开

保留两个模式：

```txt
压测模式：
  一次性加载 300 万，用来测 KLineCharts 极限。

正式工作台模式：
  首屏只加载最近 1~2 万。
  左拖按需加载。
```

不要让正式页面走压测模式。

---

### 第三刀：StoreV5 datafeed 必须有明确状态

建议新建一个清晰文件：

```txt
frontend/src/datafeed/createStoreV5KLineChartProDatafeed.ts
```

它只负责：

```txt
1. 首次加载最近 limit 根。
2. 把 StoreV5 rows 转成 KLineCharts rows。
3. 成功后明确结束 loading。
4. 左侧历史请求时再查更早窗口。
```

不要把它写在页面组件里散掉。
