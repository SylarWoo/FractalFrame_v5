```txt
  ↓
```


# 1. 总体设计原则

## 1.1 M1 是唯一原始真源头

V5 不再从 MT5 直接拉多个周期。

禁止：

```txt
MT5 → M5
MT5 → M15
MT5 → H1
MT5 → D1
```

只允许：

```txt
MT5 → M1
```

然后：

```txt
M1 → M5
M1 → M15
M1 → M30
M1 → H1
M1 → H2
M1 → H4
M1 → H8
M1 → D1
M1 → W1
M1 → MN1
```

也就是说：

```txt
Direct Store 只有 M1。
Aggregated Store 全部从 M1 派生。
```

---

## 1.2 StoreV5 只写真实 M1，不写假一分钟

MT5 返回多少条 rows，只能作为原始统计。

仓库最终写入的 rows 必须经过真实 M1 校验。

核心区别：

```txt
mt5RowsCount：
  MT5 返回了多少条 TIMEFRAME_M1 bars。

trueM1RowsCount：
  通过 UTC 22:00 锚点、第一小时完整性、60 秒连续性校验后，
  被 V5 判定为真实可用 M1 的 bars 数量。
```

正式规则：

```txt
Manifest.rowsCount = trueM1RowsCount
Manifest.mt5RowsCount = mt5RowsCount
StoreV5 只写 trueM1Rows
```

禁止：

```txt
MT5 返回多少就全写入。
用假 row 填补缺口。
把有跳跃的数据伪装成连续 M1。
把其他周期数据伪装成 M1。
```

---

## 1.3 日切锚点固定 UTC 22:00

V5 固定使用：

```txt
sessionAnchorUtc = 22:00
anchorId = UTC2200
```

原因：

```txt
UTC 22:00 = UTC+8 06:00
```

宝宝的交易日以 UTC+8 早上 6 点作为起点，所以底层统一按 UTC 22:00 对齐。

D1 区间：

```txt
[当日 UTC 22:00, 次日 UTC 22:00)
```

例如：

```txt
2025-01-01 22:00:00 UTC
  ~
2025-01-02 21:59:59 UTC
```

所有高周期聚合都继承这个锚点。

# 5. 时间制度

## 5.1 底层时间

StoreV5 底层统一保存 UTC 秒级 Unix time：

```txt
time = int Unix epoch seconds UTC
```

不要在底层保存毫秒。

前端 KLineCharts 需要毫秒时，在 datafeed 层转换：

```txt
timestamp = time * 1000
```

---

## 5.2 UTC 22:00 对齐

正式 M1 仓库的第一根 true row 必须满足：

```txt
UTC HH:mm:ss = 22:00:00
```

也就是：

```txt
firstTrueTime % 86400 == 22 * 3600
```

如果 MT5 返回数据最早不是 UTC 22:00，不直接入库。

处理方式：

```txt
从 MT5 返回 rows 中向后寻找第一根 UTC 22:00:00 的 M1 bar。
这根 bar 作为 StoreV5 Direct M1 的正式起点。
起点之前 rows 只计入 mt5RowsCount，不写入 StoreV5。
```

如果找不到 UTC 22:00 起点：

```txt
导入失败。
不写 StoreV5。
返回 error = no_utc_2200_anchor_found
```

## 8.1 首次 / Refresh 校验函数

核心函数：

```python
def validate_true_m1_rows_v1(
    rows: list[dict],
    *,

    ...
```

职责：


## 8.2 第一个 UTC 22:00 检查

检查规则：

```txt

该 row 作为 firstAnchorTime。
```

如果找不到：

## 8.3 第一小时完整性检查

从 firstAnchorTime 开始，必须检查：

```txt
22:00
22:01
22:02
...
22:59
```

也就是：

```txt
firstHourExpectedRows = 60
firstHourTrueRows = 60
```

检查方式：

```txt
firstHourRows = rows where firstAnchorTime <= time < firstAnchorTime + 3600
```

必须满足：

```txt
len(firstHourRows) == 60
并且相邻 time 差值全部为 60 秒
```

如果不满足：

```json
{
  "ok": false,
  "error": "first_hour_m1_not_continuous",
  "mt5RowsCount": 1249000,
  "firstAnchorTime": 1735682400,
  "firstHourExpectedRows": 60,
  "firstHourTrueRows": 57,
  "trueM1RowsCount": 0,
  "trueRows": []
}
```

第一小时不完整时：

```txt
不写 StoreV5。
不聚合。
不生成派生周期。
```

---

## 8.4 全段连续性检查

从 firstAnchorTime 开始扫描：

```txt
next_time - current_time == 60
```

如果出现：

```txt
next_time - current_time > 60
```

说明存在 gap。

gap report：

```json
{
  "previousTime": 1740869940,
  "nextTime": 1740870180,
  "deltaSeconds": 240,
  "missingBarsEstimate": 3
}
```

第一阶段采用严格模式：

```txt
strictContinuousMode = true
```

严格模式规则：

```txt
一旦发现第一个 gap，就截断。
只允许写入从 firstAnchorTime 到 gap 之前的连续 M1。
gap 后面的数据暂不写入正式 Direct M1。
返回 gapCount 和 firstGap。
```

如果完全无 gap：

## 9.1 Refresh 模式流程

用途：

```txt
首次建立仓库。
强制重建 M1。
修复 M1 损坏。
```

流程：

```txt
初始化 MT5。
symbol_select(symbol, True)。
删除旧 Direct M1 dataset。
删除 Manifest 中 Direct M1 cell。
copy_rates_from_pos(symbol, TIMEFRAME_M1, 0, count)。
转换 canonical rows。
执行 validate_true_m1_rows_v1。
如果 validation.ok != true，终止，不写入。
只把 validation.trueRows 传给 append_ohlcv_part_v5。
写 StoreV5 Direct M1。
更新 ManifestV5。
返回 report。
```

禁止：

```txt
append_ohlcv_part_v5(canonicalRows)
```

必须：

```txt
append_ohlcv_part_v5(validation["trueRows"])
```

---

## 9.2 Incremental 模式流程

用途：

```txt
日常补数据。
重启后继续补数据。
拉取最新 M1。
```

流程：

```txt
读取 ManifestV5 Direct M1 cell。
取得 lastTrueM1Time。
fromTime = lastTrueM1Time - overlap_bars * 60。
toTime = now UTC。
copy_rates_range(symbol, TIMEFRAME_M1, fromTime, toTime)。
转换 canonical rows。
执行 validate_incremental_true_m1_rows_v1。
如果 validation.ok != true，终止，不写入。
只把 validation.trueRows 传给 append_ohlcv_part_v5。
更新 ManifestV5。
标记所有 aggregated cell dirty=true。
返回 report。
```

增量模式禁止：

```txt
不存 MT5 position。
不依赖进程内存。
不依赖前端状态。
不跳过连续性检查。
不补假数据。
```

```txt
本批 drop_duplicates(time, keep=last)。
按 UTC year/month 分区。
读取候选 year/month 下已有 part 的 time 列。
过滤已有 time。
写 part-*.parquet。
更新 ManifestV5。
```

不要每次全局扫描全历史。

只扫描：

```txt
本次 incoming rows 涉及到的 year/month 分区。
```

Direct M1 写入时，Manifest 必须合并 `manifest_extra`：

```txt

trueM1RowsCount

gapCount
```

## 11.2 聚合规则

OHLCV 聚合：

```txt

low    = 区间 low 最小值

```

区间规则：


锚点：

```txt

```

D1：

```txt
[当日 UTC 22:00, 次日 UTC 22:00)
```


# 18. 最终系统定义

FractalFrame_v5 的数据仓库最终定义为：

```txt
StoreV5 = 一个真实 M1 源头仓 + 一组从 M1 派生的周期仓 + 一个 Manifest 状态表 + 一个 DuckDB 查询层。
```

Direct M1 的正式标准：

```txt
第一根 true row 必须对齐 UTC 22:00。
第一小时必须完整 60 根。
后续相邻 time 必须严格 60 秒。
MT5 返回 rows 要记录。
真实 M1 rows 要单独统计。
StoreV5 只写真实 M1。
```

聚合仓的正式标准：

```txt
全部从 Direct M1 派生。
全部继承 UTC2200 anchor。
全部可删除、可重建。
不能作为原始真源头。
```

最终闭环：

```txt
MT5 M1
  ↓
mt5RowsCount 统计
  ↓
trueM1Rows 校验
  ↓
UTC 22:00 对齐
  ↓
StoreV5 Direct M1
  ↓
ManifestV5
  ↓
From M1 Aggregated Store
  ↓
DuckDB Query
  ↓
Chart / Indicator / Backtest
```

一句话：

```txt
V5 不要“看起来像一分钟”的数据。
V5 只要“从 UTC 22:00 开始、第一小时完整、后续严格 60 秒递进”的真实 M1。
```
