# FractalFrame_v5 第五刀执行文案：数据仓库管理与 MT5 M1 查询拉取设计

## 1. 当前结论

V5 不应该重新发明一套散装数据方案。

V4 里已经有一条比较成熟的历史数据底座：

```txt
MT5 Import Center
  ↓
Store V2 分区 Parquet 数据仓库
  ↓
Manifest V2 可用性索引
  ↓
DuckDB 查询层
  ↓
HTTP OHLCV Query / Chart Datafeed / vectorbt export
  ↓
图表、指标、回测、策略研究
```

V5 的正确方向是：

```txt
先吸收 V4 StoreV2 的数据仓库思想
再为 V5 重写一套更干净的数据仓库管理模块
不要直接把 V4 文件原样搬进来
不要再回到“前端静态 JSON 文件就是数据源”的临时模型
```

本刀目标不是 UI，也不是实时行情，而是把 V5 的长期数据底座钉住。

---

## 2. V4 方案值得继承的部分

V4 的 Store V2 有几个核心优点，V5 应该继承：

```txt
1. M1 是直接源头。
2. 更高周期从 M1 聚合派生。
3. 数据按 provider / symbol / mode / timeframe / year / month 分区。
4. 写入使用 append-only part，不重写全历史。
5. Manifest 记录每个 dataset 的 rowsCount / partsCount / firstTime / lastTime。
6. 查询通过 DuckDB 读取 partitioned parquet。
7. 前端、回测、指标都只消费同一个历史数据真源头。
```

这套思想是对的。

尤其是 M1 数据量变大以后，不能再用旧模型：

```txt
symbol/timeframe/data.parquet
```

因为每次增量写入都要读旧文件、合并、重写整个大 parquet，M1 时间一长会越来越慢。

V5 要继续使用：

```txt
Hive-style partitioned Parquet Dataset
```

也就是：

```txt
runtime_data/market_data_store_v5/
  datasets/
    provider=mt5/
      symbol=XAUUSDm/
        mode=direct/
          timeframe=M1/
            year=2026/
              month=05/
                part-*.parquet
  manifests/
    manifest_v5.json
```

---

## 3. V4 方案需要改进的部分

V4 代码已经跑通，但 V5 不能无脑复制。

V5 要改进这些点：

### 3.1 把 MT5 拉取策略正式分层

V4 里已经出现两类 MT5 拉取方式：

```txt
copy_rates_from_pos
copy_rates_range
```

V5 要明确：

```txt
初次导入 / refresh：
  可以使用 copy_rates_from_pos 从 position 0 向历史拉一批。

增量更新：
  必须使用 Store Manifest 的 lastTime 作为持久游标，
  再用 UTC copy_rates_range(lastTime - overlap, now) 拉取。
```

原因：

```txt
MT5 position 不是长期可靠游标。
关机、终端历史缓存变化、Max bars 设置变化后，position 语义可能变化。
Manifest lastTime 才是本地仓库自己的稳定游标。
```

所以 V5 的主语义是：

```txt
增量不记 position。
增量只认 time cursor。
```

---

### 3.2 Manifest 要从“文件索引”升级为“数据资产状态表”

V4 Manifest 已经记录 rowsCount / firstTime / lastTime 等信息。

V5 建议进一步明确 Manifest cell：

```json
{
  "provider": "mt5",
  "symbol": "XAUUSDm",
  "mode": "direct",
  "timeframe": "M1",
  "baseTimeframe": null,
  "rootPath": "datasets/provider=mt5/symbol=XAUUSDm/mode=direct/timeframe=M1",
  "rowsCount": 1234567,
  "partsCount": 38,
  "firstTime": 1710000000,
  "lastTime": 1770000000,
  "lastSuccessfulPullAt": "2026-05-19T00:00:00Z",
  "lastPullMethod": "copy_rates_range",
  "lastPullFromTime": 1769990000,
  "lastPullToTime": 1770000000,
  "lastPullRawRows": 1200,
  "lastPullAddedRows": 180,
  "lastPullDuplicateRows": 1020,
  "dirty": false,
  "compactRecommended": false,
  "schemaVersion": "5.0.0",
  "createdAt": "...",
  "updatedAt": "..."
}
```

这样以后右侧 Data Center 抽屉可以直接显示：

```txt
这个品种有没有数据
M1 最早到哪
M1 最新到哪
上次拉取是否成功
这次新增多少 rows
是否建议 compact
```

---

### 3.3 Writer 要继续 append-only，但要支持按月去重

V5 写入原则：

```txt
普通导入：append part。
同批数据：按 time 去重，keep last。
候选月份：读取已有 part 的 time 列，过滤重复 time。
长期维护：compact job 合并月分区，最终去重。
```

不要在每次导入时全局扫描全历史。

正确边界是：

```txt
导入层：决定拉取窗口。
writer：负责本批与候选分区去重。
compact：负责月分区归并、最终整理。
query：只读，不改变仓库。
```

---

### 3.4 DuckDB 查询层要成为唯一后端查询入口

V5 不要让前端直接读一堆 JSON 文件。

正式模型：

```txt
前端请求 /api/ohlcv/query
  ↓
后端读 manifest_v5.json
  ↓
定位 dataset root
  ↓
按 time_from / time_to 收集 year/month part files
  ↓
DuckDB read_parquet(files, hive_partitioning=true)
  ↓
返回 timestamp/open/high/low/close/volume
```

这样未来：

```txt
图表加载
指标计算
回测导出
Data Center 状态显示
```

都吃同一套数据事实源。

## 4. 本刀要加的核心能力：MT5 M1 查询拉取

这里的“MT5 M1 查询拉取”不是前端查询，也不是实时订阅。

它是一个后端数据导入/同步能力：

```txt
给定 symbol = XAUUSDm
给定 timeframe = M1
读取 Store V5 Manifest 中已有 M1 数据边界
调用 MT5 Python API 拉取缺失/最新 M1 bars
转换为 canonical OHLCV
写入 Store V5 partitioned parquet
更新 manifest_v5.json
返回 pull report
```

## 6. MT5 M1 Pull Service 语义

新增文件建议：

```txt
python/data_warehouse/mt5/mt5_m1_pull_service_v1.py
```

核心函数：

```python
def pull_mt5_m1_to_store_v5(
    *,
    symbol: str,
    count: int = 500_000,
    import_mode: str = "incremental",  # incremental | refresh
    store_root: str | Path | None = None,
    overlap_bars: int = 1000,
    mt5_module: Any | None = None,
) -> dict[str, Any]:
    ...
```

### 6.1 refresh 模式

```txt
删除旧 direct/M1 dataset
从 MT5 copy_rates_from_pos(symbol, TIMEFRAME_M1, 0, count) 拉取
写入 Store V5
更新 Manifest
```

用于：

```txt
首次压测
强制重建
历史数据损坏后重拉
```

### 6.2 incremental 模式

```txt
读取 manifest_v5 的 lastTime
fromTime = lastTime - overlap_bars * 60
toTime = now UTC
调用 copy_rates_range(symbol, TIMEFRAME_M1, fromTime, toTime)
转换 rows
只保留 time > lastTime 的新 rows
写入 Store V5
更新 Manifest
```

用于：

```txt
每天补数据
重启后继续补数据
Data Center 点击“同步最新 M1”
```

### 6.3 为什么 incremental 必须 range

MT5 官方文档说明：

```txt
copy_rates_range 返回指定 date_from 到 date_to 范围内的 bars。
返回字段包括 time/open/high/low/close/tick_volume/spread/real_volume。
MT5 bar 时间是 UTC，Python datetime 也应使用 UTC。
终端可用 bars 受 Max bars in chart/history 设置影响。
```

所以 V5 的增量策略应该以 UTC 时间窗口为核心。

## 7. canonical OHLCV 数据格式

Store V5 内部保存秒级 epoch time：

```json
{
  "time": 1716000000,
  "open": 2350.12,
  "high": 2352.4,
  "low": 2348.7,
  "close": 2351.22,
  "volume": 595,
  "provider": "mt5",
  "symbol": "XAUUSDm",
  "timeframe": "M1",
  "source": "mt5_terminal",
  "ingestedAt": "2026-05-19T00:00:00Z"
}
```

前端 KLineCharts 输出时再转换为毫秒：

```txt
timestamp = time * 1000
```

不要在仓库底层保存毫秒 timestamp。

原因：

```txt
MT5 原生 time 是秒。
DuckDB 时间过滤用秒更直接。
聚合 M5/M15/H1 更简单。
前端适配层转换一次即可。
```

### 11.2 增量验收

```txt
python -m python.data_warehouse.scripts.pull_mt5_m1_to_store_v5 --symbol XAUUSDm --mode incremental
```

通过标准：

```txt
不会全量重写旧数据

provider = store_v5_duckdb
storeVersion = v5
```

## 12. 给 V5 的最终设计判断

V5 的数据仓库主线应该是：

```txt
MT5 M1 Puller = 采集器
Store V5 = 本地历史数据真源头
Manifest V5 = 数据资产状态表
DuckDB Query = 唯一查询引擎
Aggregator = 从 M1 派生 M5/M15/H1 等周期
Data Center = UI 运维入口
Chart Datafeed = 只消费，不写仓库
```

一句话：

```txt
V5 不要做“图表项目里顺手存点数据”。
V5 要做“本地市场数据仓库 + 图表工作台”。
```
