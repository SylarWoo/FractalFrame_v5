# 执行指令：FractalFrame_v5 七项优化完整执行计划

> 本文档是后续代码优化任务的执行总纲。目标是让执行者按固定顺序推进，不在长任务中遗漏细节，也不因为局部重构破坏当前已经跑通的功能。

## 一、总目标

在不重做 UI、不改变现有业务行为、不更换技术栈的前提下，把当前项目从“功能能跑”整理到“模块边界清楚、状态来源集中、测试和质量门槛稳定”的状态。

本轮优化覆盖 7 个方向：

1. 拆分过大的 `RightDrawer.tsx`
2. 统一 `localStorage` 状态持久化
3. 抽离顶部栏状态逻辑
4. 按模块拆分前端 CSS
5. 增加前端关键逻辑测试
6. 拆分 Python 后端大文件
7. 增加统一质量检查命令

## 二、执行总原则

1. 每次只做一个明确阶段，不把多个大方向混在同一次提交里。
2. 优先做无行为变化的拆分，再做功能性修复。
3. 拆文件时保留原有导出名、HTTP path、localStorage key、按钮文案和用户可见行为。
4. 不删除 `runtime_data` 中的用户数据。
5. 不做大范围视觉重设计，只做结构整理。
6. 遇到已有改动时，先判断是否与当前任务相关，不要回滚用户改动。
7. 每完成一个阶段，必须运行对应验证命令，并把结果记录到本文末尾“执行记录”。

## 三、当前重点风险

### 1. 前端大文件风险

重点文件：

```txt
frontend/src/workbench/rightDrawer/RightDrawer.tsx
frontend/src/workbench/rightDrawer/RightDrawer.css
frontend/src/workbench/settings/SettingsPanel.tsx
frontend/src/workbench/chart/ChartCoreHost.tsx
frontend/src/services/mt5/mt5SymbolsApi.ts
frontend/src/workbench/AppShell.tsx
frontend/src/workbench/topbar/TopBar.tsx
```

风险：UI、业务状态、持久化、事件处理混在一起；后续修改容易产生连带回归；CSS 选择器容易互相影响。

### 2. 状态分散风险

`localStorage` 读写曾分布在多个文件：

```txt
frontend/src/workbench/topbar/TopBar.tsx
frontend/src/workbench/AppShell.tsx
frontend/src/workbench/settingsSymbolState.ts
frontend/src/workbench/mt5DataCenter/storeV5Persistence.ts
frontend/src/workbench/rightDrawer/RightDrawer.tsx
```

风险：key 重复或命名不一致；JSON fallback 行为不一致；跨组件同步事件容易遗漏。

### 3. 后端大文件风险

重点文件：

```txt
scripts/mt5_symbols_server.py
```

风险：路由、任务状态、SSE、StoreV5 操作混在一个入口文件中；后续增加接口时难以测试；任务状态和响应格式容易被误改。

## 五、阶段 1：拆分 `RightDrawer.tsx`

目标：让 `RightDrawer.tsx` 只负责右侧抽屉外壳、打开关闭、宽度和子面板挂载，不再承载大量业务细节。

建议结构：

```txt
frontend/src/workbench/rightDrawer/
  RightDrawer.tsx
  RightDrawerShell.tsx
  RightDrawer.css

frontend/src/workbench/mt5DataCenter/
  Mt5DataCenterPanel.tsx
  WatchlistTable.tsx
  StoreV5Panel.tsx
  useStoreV5Jobs.ts
  useWatchlistRealtimeTicks.ts
  storeV5Persistence.ts
  storeV5StatusFormat.ts
  types.ts
```

执行顺序：先抽无状态纯函数和类型；再抽 watchlist 表格组件；再抽 StoreV5 操作区组件；再抽 MT5 数据中心面板；最后让 `RightDrawer.tsx` 只装配右侧抽屉、设置面板和 MT5 面板。

禁止事项：不改按钮文案；不改接口调用顺序；不改 localStorage key；不改右侧抽屉默认宽度和开关行为。

验收标准：Settings 面板正常打开关闭；MT5 数据中心正常显示；watchlist 表格、列宽、选中状态正常；StoreV5 拉取、检查、聚合、删除等入口仍可操作；`RightDrawer.tsx` 行数明显下降；`npm run build` 通过。

## 六、阶段 2：统一 `localStorage` 状态持久化

目标：把散落的 storage key、读写函数、JSON fallback 和跨组件通知集中管理。

建议结构：

```txt
frontend/src/workbench/persistence/
  storageKeys.ts
  jsonStorage.ts
  usePersistentState.ts
  workbenchEvents.ts
```

执行顺序：新建 `storageKeys.ts` 集中导出 key；新建 `jsonStorage.ts` 封装 `readJson`、`writeJson`、`readString`、`writeString`、`readBooleanFlag`、`writeBooleanFlag`；新建 `workbenchEvents.ts` 集中管理自定义事件；逐步替换 `TopBar.tsx`、`AppShell.tsx`、`storeV5Persistence.ts`、`settingsSymbolState.ts`。

禁止事项：不改任何现有 key 字符串的值；不改 JSON 数据结构；不删除历史 fallback；不让某个模块直接依赖另一个 UI 组件。

验收标准：刷新页面后右侧抽屉状态保持正常；顶部品种和周期选择保持正常；设置页选项保持正常；MT5 数据中心 watchlist 和状态保持正常；`npm run build` 通过。

## 七、阶段 3：抽离顶部栏状态逻辑

目标：让 `TopBar.tsx` 主要负责渲染，状态读取、周期合并、共享选择同步交给 hook。

建议结构：

```txt
frontend/src/workbench/topbar/
  TopBar.tsx
  TopBar.css
  useShortcutMenuState.ts
  topbarPeriodUtils.ts
```

执行顺序：把 `periodOrder`、`periodToChartPeriod` 移到 `topbarPeriodUtils.ts`；抽离 watchlist symbols、shortcut periods、shared selection 的读取逻辑；新建 `useShortcutMenuState.ts` 管理 selectedSymbol、activePeriod、open 状态并监听 storage 和自定义事件；`TopBar.tsx` 只接收 hook 返回值并渲染按钮。

禁止事项：不改当前顶部栏视觉样式；不恢复已取消的下拉三角；不改周期顺序；不改打开图表时传入 `onOpenChart` 的参数结构。

验收标准：顶部品种下拉可打开、可选择；周期按钮正常切换；选择品种和周期后图表正常加载；刷新后共享选择状态正常恢复；`npm run build` 通过。
