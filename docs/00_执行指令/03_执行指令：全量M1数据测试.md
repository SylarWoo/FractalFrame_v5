对，宝宝这个路线是对的：**不用假 20W，直接从 MT5 抽 M1 全量真实数据**。
但要注意一句：**“M1 全量不卡”只能证明图表核心和基础数据渲染能扛住；后面大改 UI 以后不一定绝对不卡**。如果 UI 改动只是外壳、菜单、样式、布局，基本不会影响 K 线拖动；但如果后面加很多实时联动、复杂指标、画线对象、React 状态监听，那仍然可能把性能拖下来。所以这次压测要把“基础图表性能基线”钉死。

# FractalFrame_v5 第三刀执行文案：MT5 M1 全量数据压测 KLineChart Pro

## 1. 当前背景

FractalFrame_v5 已经完成 KLineChart Pro 的最小接入验证。

当前已确认：

```txt
KLineChart Pro 页面可以正常显示
Mock Datafeed 可以正常喂 K线
MA / VOL / 基础工具栏 / 画线工具初步可用
```

现在不要继续做 1000 / 2000 根小数据测试，因为 lightweight-charts 在这个量级本来也很舒服。当前真正要验证的是：

```txt
KLineChart Pro / KLineCharts 是否能承载 MT5 M1 全量历史数据
并且在左翻、右翻、缩放、画线、指标开启时不卡
```

所以第三刀目标不是“接入 MT5 实时行情”，而是：

```txt
从 MT5 抽出指定品种的 M1 全量历史数据
保存成本地静态数据文件
前端一次性加载到 KLineChart Pro
进行大数据量交互压测
```

## 2. 核心目标

```txt
从 MT5 导出全部 M1 K线
将数据转换成 KLineCharts 需要的 OHLCV 格式
前端加载这份全量数据
测试 KLineChart Pro 在真实 M1 全量数据下是否卡顿
```

MT5 Python 官方提供 `copy_rates_from_pos(symbol, timeframe, start_pos, count)`，可以按品种、周期、起始位置和数量拉取 K 线；返回字段包含 `time/open/high/low/close/tick_volume/spread/real_volume`。([mql5.com][1])

KLineCharts 的 K 线数据格式需要 `timestamp/open/high/low/close/volume` 等字段，且 `timestamp` 使用毫秒。([GitHub][2])

---

## 3. 关键原则

### 3.1 不要生成假 20W 数据

不要自己随机生成 20W mock 数据。

原因：

```txt
假数据只能测试数量压力
测不出真实 MT5 数据里的时间断层、交易时段、跳空、volume 分布、小数位、黄金品种特征
```

本刀必须使用 MT5 真实 M1 数据。

---

### 3.2 不要一边连 MT5 一边测图表

本刀不要直接做前端 API 实时请求 MT5。

正确方式是：

```txt
MT5 → Python 导出 → 本地静态 JSON/Parquet → 前端 fetch → KLineChart Pro
```

这样能把问题切干净：

```txt
如果卡：
就是图表加载/渲染/交互性能问题

如果不卡：
说明 KLineChart Pro 基础大数据承载能力过关
```

不要让 MT5 连接、接口延迟、Python 服务、网络传输一起混进压测结果里。

---

### 3.3 先测基础图表，不急着改 UI

这刀只测核心图表承载能力。

不要在这刀同时做：

```txt
复杂 UI 改版
侧边栏重构
多品种切换
实时推送
DuckDB
无限历史加载
增量更新
复杂画线系统改造
```

注意：

```txt

Max bars in chart
Max bars in history
```

尽量调大。

## 6. 导出数据格式

MT5 返回的 `time` 是秒级 Unix 时间。

KLineCharts 要毫秒 timestamp。

所以转换规则：

```txt
timestamp = int(row["time"]) * 1000
open = float(row["open"])
high = float(row["high"])
low = float(row["low"])
close = float(row["close"])
volume = int(row["tick_volume"])
```

最终 JSON 单条格式：

```json
{
  "timestamp": 1716000000000,
  "open": 2350.12,
  "high": 2352.4,
  "low": 2348.7,
  "close": 2351.22,
  "volume": 595
}
```

## 11. 如果不卡，是否代表以后大改 UI 也不会卡？

结论要谨慎：

```txt
如果 M1 全量数据 + MA + VOL + 少量画线都不卡，
说明 KLineChart Pro / KLineCharts 的核心渲染性能基本过关。
后面做普通 UI 外壳改版，大概率不会让 K线本体变卡。
```

但不能说“以后怎么改都不会卡”。

因为后续如果加入这些东西，仍然可能卡：

```txt
大量 React state 绑定到鼠标移动
每次 crosshair move 都 setState
每次拖动都触发全局 store 更新
大量 DOM overlay 跟随 K线移动
几十个指标同时实时重算
几百上千条画线对象
每次缩放都重新 fetch / 重新 setData
每次 viewport 改变都重建 chart 实例
```

所以本刀通过后，只能得出这个结论：

```txt
图表核心大数据承载能力通过。
后续 UI 改造可以继续，但必须遵守性能隔离原则：
UI 外壳不要绑死图表高频事件。
图表实例不要被 React 反复重建。
指标尽量预计算或增量计算。
画线对象数量要分阶段压测。
```
